# Decision Log

## Decision 001: Use Python 3.11

### Context

PyTorch와 데이터 분석 패키지를 안정적으로 함께 사용해야 한다.

### Alternatives

1. Python 3.10
2. Python 3.11
3. 최신 Python 버전

### Decision

Python 3.11을 사용한다.

### Reason

최신 문법보다 패키지 호환성과 프로젝트 재현성을 우선한다.

### Risk

장기 프로젝트 중 일부 최신 패키지가 더 높은 Python 버전을
요구할 가능성이 있다.

### Revisit condition

핵심 패키지가 Python 3.11 지원을 종료하거나,
필수 기능이 상위 버전에서만 제공될 때 재검토한다.

## Decision 002: CUDA 기반 PyTorch 환경 사용

### 상황

개발 장비에 NVIDIA RTX 3050 GPU가 설치되어 있으며, 향후 PyTorch 기반 신경망 모델과 시계열.공간 모델을 실험할 계획이다.

### 검토한 선택지

1. CPU 전용 PyTorch
2. CUDA 기반 GPU PyTorch
3. 초기에는 CPU를 사용하고 이후 GPU 환경으로 전환

### 결정

CUDA 12.6 기반 PyTorch 환경을 초기 개발환경으로 사용한다.

### 근거
- 현재 장비에서 `torch.cuda.is_available()`이 `True`로 확인됐다.
- 동일한 Conda 환경에서 데이터 처리와 GPU 학습을 모두 수행할 수 있다.
- 이후 MLP, Temporal Fusion Transformer, Graph Neural Network 등을 비교할 때 환경을 다시 구축할 필요가 없다.
- GPU 환경을 먼저 검증해 CUDA 호환성 문제를 초기 단계에서 발견할 수 있다.

### 확인 결과
- PyTorch 버전: `2.12.1+cu126`
- CUDA 사용 가능 여부: `True`
- 기본 텐서 연산 결과: `11.0`

### 위험 요소
- CUDA 패키지는 CPU 버전보다 설치 용량이 크다.
- PyTorch, torchvision, CUDA 버전 간 호환성이 맞지 않으면 실행 오류가 발생할 수 있다.
- GPU 메모리가 부족한 경우 배치 크기 조정이나 CPU 대체 실행이 필요하다.

### 재검토 조건
다음 상황이 발생하면 환경 구성을 다시 검토한다.
- PyTorch와 CUDA 버전 충돌
- GPU 메모리 부족으로 학습이 반복적으로 실패
- 클라우드 환경과 로컬 환경의 재현성 차이 발생
- 운영 파이프라인에서 CPU 실행이 더 경제적인 것으로 판단됨

## Decision 003: 최초 분석 기간을 2021~2025년으로 제한

### 상황

서울시 상권분석서비스는 2024년부터 공간 단위를 변경했고,
2026년 7월 3일부터 2021년 이후 자료만 제공할 예정이다.

### 검토한 선택지

1. 2019~2025년 전체 사용
2. 2021~2025년 사용
3. 2024~2025년만 사용

### 결정

최초 분석에서는 2021~2025년 자료를 사용한다.

### 근거

- 향후에도 다시 수집하고 검증할 수 있는 기간을 우선해 프로젝트 재현성을 높인다.
- 2019년~2020년을 추가하면 관측 기간은 늘어나지만, 현재 제공 범위와 원천 스키마를 별도로 복원하고 검증해야 한다.
- 2024년 이후 자료만 사용하면 공간 정의는 상대적으로 단순해지지만, 시계열 학습과 전후 변화 분석에 필요한 기간이 지나치게 짧아진다.
- 2021년~2025년은 분석 가능한 기간을 확보하면서도, 2024년 공간 단위 변경을 명시적으로 검증 과제로 포함할 수 있는 절충안이다.
- 다만 2021년~2023년과 2024년~2025년을 동일한 공간 단위라고 가정하지 않고, 코드와 상권 구성의 연속성을 먼저 검사한다.

### 예상되는 위험

- 5년의 분기 자료만으로는 장기 경기 사이클과 다양한 외부 충격을 충분히 학습하기 어렵다.
- 2024년 공간 단위 변경으로 같은 상권 코드의 의미가 달라지거나, 코드가 생성·소멸·분할·통합됐을 수 있다.
- 전체 기간을 단순 결합하면 실제 변화가 아니라 공간 정의 변경을 모델이 패턴으로 학습할 수 있다.
- 2024년 이후만 별도로 분석할 경우 학습 표본과 시계열 길이가 부족할 수 있다.

### 검증 방법

- 연도별·분기별 행 수, 고유 상권 수와 업종 수를 비교한다.
- 2023년과 2024년 사이의 상권 코드 교집합, 신규 코드, 소멸 코드를 계산한다.
- 상권 코드와 상권 이름의 대응 관계가 시점별로 달라지는지 확인한다.
- 점포 수와 주요 변수의 분포가 2024년을 경계로 급변하는지 비교한다.
- 전체 2021~2025년 모델과 2024~2025년 전용 모델을 각각 검증하여 성능과 안정성을 비교한다.
- 공간 정의가 직접 비교되지 않는 경우, 매핑 테이블을 구성하거나 기간을 분리해 모델링한다.

### 재검토 조건

- 신뢰할 수 있는 2019년~2020년 원본 스냅샷과 스키마·공간 단위 매핑을 확보했을 때
- 2024년 전후 상권 코드와 공간 정의의 연속성이 낮아 전체 기간 결합이 부적절하다고 확인됐을 때
- 2024년 이후 표본만으로도 충분한 학습·검증이 가능하다고 판단됐을 때
- 모델 오차 분석에서 기간 정의 차이가 주요 실패 원인으로 확인됐을 때

## Decision 004: 내부 표준 컬럼명을 영어 snake_case로 관리

### 상황

2021년~2024년 점포 데이터는 한국어 헤더를 사용하지만,
2025년 데이터는 같은 의미의 컬럼을 영문 약어 헤더로 제공한다.
원본 컬럼명을 그대로 사용하면 연도별로 다른 이름을 처리해야 하며,
후속 조인·집계·검증 코드가 특정 연도의 원본 스키마에 종속된다.

### 검토한 선택지

1. 한국어 컬럼명을 내부 표준으로 사용
2. 2025년의 공식 영문 약어를 그대로 사용
3. 의미가 명확한 영어 snake_case를 별도로 정의

### 결정

의미가 명확한 영어 `snake_case`를 내부 표준 컬럼명으로 사용한다.

### 근거

- 원본 제공기관의 연도별 컬럼명 변경과 내부 분석 코드를 분리할 수 있다.
- 한국어 헤더와 2025년 영문 약어를 각각 alias로 관리하고, 후속 코드는 하나의 표준 이름만 사용하게 할 수 있다.
- 영문 약어를 그대로 사용하면 이름이 짧지만 의미가 불분명하므로, 전체 단어를 사용한 표준명이 코드 리뷰와 문서화에 유리하다.
- Python과 pandas의 일반적인 명명 규칙과 일치해 함수 인자, 설정 파일, 테스트와 모델 입력 컬럼을 일관되게 관리할 수 있다.
- 새 연도의 원본 헤더가 바뀌더라도 alias 설정과 데이터 계약만 수정하면 후속 파이프라인을 유지할 수 있다.

### 예상되는 위험

- 번역이 부정확하면 원본 컬럼의 업무적 의미가 표준명에서 왜곡될 수 있다.
- 매핑표에서 원본 컬럼이 누락되면 필수 표준 컬럼이 생성되지 않을 수 있다.
- 서로 다른 원본 컬럼을 동일한 표준명으로 매핑하면 컬럼 충돌이나 값 덮어쓰기가 발생할 수 있다.
- 원본 컬럼명은 같지만 실제 정의나 단위가 변경된 경우, 이름 정규화만으로 의미 변화를 숨길 수 있다.
- 원본 데이터 사전과 내부 표준명 사이의 대응 관계를 문서화하지 않으면 추적성이 낮아진다.

### 검증 방법

- 연도별 원본 컬럼이 하나의 표준 컬럼에 정확히 대응하는지 alias 매핑 테스트를 작성한다.
- 모든 필수 컬럼이 존재하면 정상 통과하고, 하나라도 누락되면 즉시 오류가 발생하는지 검사한다.
- 원본 컬럼 수와 정규화 컬럼 수를 비교하고, 중복된 표준 컬럼명이 없는지 확인한다.
- 2021년~2025년 정규화 결과의 컬럼 순서와 스키마 서명이 동일한지 검사한다.
- 오류 메시지에 대상 파일, 연도, 누락된 컬럼과 실제 발견된 컬럼을 포함한다.
- 원본 데이터 사전과 내부 표준 컬럼의 의미·단위·코드 정의를 함께 관리한다.

### 재검토 조건

- 제공기관이 여러 연도에 걸쳐 안정적이고 의미가 명확한 공식 표준 컬럼명을 제공할 때
- 내부 표준명의 번역이 원본 의미를 충분히 표현하지 못한다고 확인됐을 때
- 다른 데이터 소스 또는 외부 시스템과의 연동에서 별도의 표준 명명 규칙이 필요해질 때
- 컬럼명보다 스키마 버전과 의미 정의를 중심으로 관리하는 구조로 전환할 때
